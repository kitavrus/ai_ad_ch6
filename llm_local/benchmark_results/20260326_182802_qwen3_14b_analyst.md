### C4 Context диаграмма (уровень контекста):  
**Система:** Сервис обработки платежей  
**Внешние системы:**  
1. **Платежный шлюз** (Payment Gateway) – обработка транзакций и взаимодействие с банками.  
2. **Банк** (Bank) – обработка реальных платежей (снятие/зачисление средств).  
3. **Мерчант** (Merchant) – инициатор платежа (например, интернет-магазин).  
4. **Клиент** (Customer) – пользователь, осуществляющий платеж.  
5. **Система обнаружения мошенничества** (Fraud Detection System) – проверка транзакции на подозрительность.  
6. **База данных транзакций** (Transaction Database) – хранение информации о платежах.  

**Данные, обмениваемые между системами:**  
- **Мерчант → Сервис обработки платежей:** Запрос на платеж (Payment Request).  
- **Сервис обработки платежей → Система обнаружения мошенничества:** Данные транзакции (Transaction Data).  
- **Система обнаружения мошенничества → Сервис обработки платежей:** Результат проверки (Approval/Decline).  
- **Сервис обработки платежей → Платежный шлюз:** Запрос на авторизацию (Payment Authorization Request).  
- **Платежный шлюз → Банк:** Обработка платежа (Process Payment).  
- **Банк → Платежный шлюз:** Подтверждение транзакции (Transaction Confirmation).  
- **Платежный шлюз → Сервис обработки платежей:** Результат авторизации (Authorization Result).  
- **Сервис обработки платежей → База данных транзакций:** Сохранение данных (Store Transaction).  
- **Сервис обработки платежей → Мерчант/Клиент:** Подтверждение платежа (Payment Confirmation).  

---

### Пользовательские истории:  
1. **As a Merchant, I want to send a payment request to the payment service, so that the customer can complete the transaction.**  
   - **Acceptance criteria:**  
     - Merchant отправляет данные транзакции (сумма, метод оплаты, идентификатор клиента).  
     - Сервис обработки платежей принимает запрос и проверяет его на соответствие требованиям.  

2. **As a Customer, I want to receive a confirmation of the payment after it is processed, so that I know the transaction was successful.**  
   - **Acceptance criteria:**  
     - Сервис отправляет уведомление клиенту (по email или в приложение) после завершения транзакции.  

3. **As a Fraud Detection System, I want to analyze transaction data in real time, so that suspicious payments are blocked before processing.**  
   - **Acceptance criteria:**  
     - Сервис передает данные транзакции в систему обнаружения мошенничества.  
     - Система возвращает результат проверки (разрешить/отказать) в течение 1–2 секунд.  

---

### Бизнес-процессы (BPMN-подобный текст):  
1. **Инициация платежа** (Merchant → Payment Service):  
   - **Актор:** Мерчант.  
   - **Гейтвей:** Проверка валидности данных (сумма > 0, метод оплаты поддерживается).  
   - **Результат:** Если данные недействительны → ошибка. Если данные валидны → переход к проверке на мошенничество.  

2. **Проверка на мошенничество** (Payment Service → Fraud Detection):  
   - **Гейтвей:** Проверка IP-адреса, истории транзакций, соответствия данных.  
   - **Результат:** Если подозрительная → блокировка. Если не подозрительная → продолжение.  

3. **Обработка платежа** (Payment Service → Payment Gateway → Bank):  
   - **Гейтвей:** Передача данных банку для обработки (списание средств).  
   - **Результат:** Получение подтверждения от банка (успех/ошибка).  

4. **Хранение данных** (Payment Service → Transaction Database):  
   - **Гейтвей:** Сохранение информации о транзакции (ID, сумма, статус).

## 2. [coding]

**Вопрос:** Напиши функцию на Python для разбора JWT-токена без сторонних библиотек.

**Статистика:** время `83.7s` | символов `2248` | скорость `27 сим/с`

**Ответ:**

```python
import base64
import json

def parse_jwt(token):
    """
    Разбирает JWT-токен на части: заголовок, полезная нагрузка, подпись.
    Не проверяет подпись и не валидирует токен.
    
    Args:
        token (str): JWT-токен в формате "header.payload.signature"
    
    Returns:
        dict: Словарь с ключами 'header', 'payload', 'signature'
    
    Raises:
        ValueError: Если токен некорректный
    """
    if not token or '.' not in token:
        raise ValueError("Некорректный JWT-токен")
    
    parts = token.split('.')
    if len(parts) != 3:
        raise ValueError("JWT-токен должен содержать 3 части")
    
    # Функция для декодирования Base64 URL
    def urlsafe_b64decode(s):
        # Заменяем недопустимые символы для Base64
        s = s.replace('-', '+').replace('_', '/')
        # Добавляем недостающие '='
        pad = len(s) % 4
        if pad:
            s += '=' * (4 - pad)
        return base64.b64decode(s).decode('utf-8')
    
    try:
        header = json.loads(urlsafe_b64decode(parts[0]))
        payload = json.loads(urlsafe_b64decode(parts[1]))
    except (json.JSONDecodeError, base64.binascii.Error) as e:
        raise ValueError(f"Ошибка декодирования: {str(e)}")
    
    return {
        'header': header,
        'payload': payload,
        'signature': parts[2]
    }
```

### Основные моменты:
1. **Формат JWT**: JWT состоит из трёх частей, разделённых точками: `header.payload.signature`.
2. **Base64 URL-кодирование**: Использует замену символов `-` на `+` и `_` на `/`, а также добавляет недостающие символы `=`.
3. **Декодирование**: Выполняется через стандартную библиотеку `base64`.
4. **Проверка ошибок**: Обрабатывается некорректный формат токена и ошибки декодирования.

### Ограничения:
- **Не проверяет подпись**: Функция не проверяет подлинность токена.
- **Не валидирует данные**: Не проверяет содержимое заголовка и полезной нагрузки на соответствие стандартам.

## 3. [general]

**Вопрос:** Объясни теорему CAP в трёх предложениях.

**Статистика:** время `25.8s` | символов `529` | скорость `20 сим/с`

**Ответ:**

Теорема CAP утверждает, что в распределённых системах невозможно одновременно обеспечить все три свойства: согласованность (Consistency), доступность (Availability) и толерантность к разделению (Partition tolerance). Система должна выбрать два из трёх, например, приоритет может быть отдан согласованности и толерантности к разделению, тогда доступность будет ограничена. Это ограничение критично для проектирования баз данных и распределённых приложений, где выбор между компромиссами определяет производительность и надёжность.
