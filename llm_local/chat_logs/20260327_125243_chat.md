### Что нельзя тестировать напрямую:  
- **Архитектурные решения** (например, выбор базы данных или сервера) — их можно проверить через **рефакторинг** и **code review**.  
- **Связи между системами** (например, интеграция с картами) — тестируются через **API-тестирование**.  
- **Бизнес-логика** (например, алгоритмы расчёта доставки) — тестируются через **модульное тестирование**.  

---

### Рекомендации по тестированию:  
- **Функциональные тесты**: Проверка создания заказов, оплаты, отслеживания доставки.  
- **Регрессионные тесты**: После обновлений меню или правил доставки.  
- **Граничные случаи**:  
  - Заказы с минимальной суммой (например, 0 руб).  
  - Отсутствие курьеров в определённой зоне.  
  - Ошибки оплаты (например, недостаточно средств).  
- **Автоматизация**:  
  - **Unit-тесты**: Для логики расчёта стоимости доставки.  
  - **E2E-тесты**: Через **Selenium** или **Playwright** для проверки полного цикла "заказ → доставка".  
  - **API-тесты**: Через **Postman** или **Karate** для проверки интеграций с ресторанами и платежными системами.
