# Примеры использования /plan on и /plan builder

## 1. `/plan on` — пример сессии

**Сценарий:** написание REST API с инвариантами.

```
python script.py

> /plan on
[Plan mode: ON | инвариантов: 0 | max_retries: 3]
[Профиль: default | (нет настроек)]
[Введите описание задачи:]

> Написать REST API для управления задачами

[Хотите добавить инварианты? (да/нет):]

> да

[Добавляйте инварианты через `/invariant add <текст>`. Введите 'готово' когда закончите.]

> /invariant add Всегда отвечай на русском
[Инвариант добавлен #1: Всегда отвечай на русском]

> /invariant add Не используй SQL напрямую, только ORM
[Инвариант добавлен #2: Не используй SQL напрямую, только ORM]

> готово

[Запускаю диалог планирования...]

# Агент делает первый LLM-вызов с контекстом задачи: "Написать REST API для управления задачами"

> Как реализовать endpoint для создания задачи?

# Агент строит промпт: Роль + Профиль + Текущее состояние + Инварианты
# Генерирует черновик ответа
# Валидирует черновик против каждого инварианта отдельным вызовом LLM
# Если нарушение → ретрай (до 3 раз)
# Парсит блоки Response и State Update из финального ответа

**Response:**
Для создания задачи используй POST /tasks с телом {"title": "...", "description": "..."}.
В FastAPI это выглядит так:

    @router.post("/tasks", response_model=TaskOut)
    def create_task(data: TaskIn, db: Session = Depends(get_db)):
        task = Task(**data.dict())
        db.add(task)
        db.commit()
        return task

**State Update:**
last_topic: task creation endpoint

> Как добавить валидацию?
# → следующий запрос тоже проходит через агентский цикл
# State Update из предыдущего ответа сохранён в рабочую память
# → виден в системном промпте следующего запроса

**Response:**
Для валидации добавь Pydantic-схему:

    class TaskIn(BaseModel):
        title: str = Field(..., min_length=1, max_length=200)
        description: str = ""

**State Update:**
last_topic: input validation

> /invariant list
[Инварианты:
  1. Всегда отвечай на русском
  2. Не используй SQL напрямую, только ORM]

> /agent status
[Agent mode: ON | retries: 3 | инвариантов: 2]
```

## 5. Восстановление после сбоя

**Проблема:** при Ctrl+C или потере соединения во время `/plan builder` `active_task_id`
мог потеряться, если пользователь не отправил ни одного обычного сообщения после создания плана.

**Решение (автоматическое с текущей версии):** сессия сохраняется в двух дополнительных точках:
- сразу после создания плана (`/task new`)
- после завершения каждого шага builder

```
> /task new "Разработать REST API"
[План создан: Разработать REST API]
# ← сессия сохранена, active_task_id зафиксирован

> /plan builder

[Builder: выполняю шаг 1. Описать модель данных...]
[Builder: шаг 1 DONE]
# ← сессия сохранена

# --- Ctrl+C / потеря интернета ---

python script.py --resume
# ← active_task_id восстановлен автоматически

> /plan builder
# builder видит незавершённые шаги и продолжает с шага 2
[Builder: выполняю шаг 2. Реализовать CRUD endpoints...]
```

Если восстановление по какой-то причине не сработало — ручной вариант:

```
> /task list          # найти нужный task_id
> /task load <id>     # восстановить active_task_id вручную
> /plan builder       # продолжить выполнение
```

## Ключевые команды

| Команда | Описание |
|---------|----------|
| `/plan on` | Интерактивный старт: запрашивает задачу и инварианты |

## 8. Демо-сценарии системы инвариантов

Инварианты — жёсткие ограничения, которым должен следовать каждый ответ агента.
При нарушении запускается авто-ретрай (до `max_retries` раз), затем — уточняющий вопрос к пользователю.

Все сценарии воспроизводимы: запустите `python script.py` и повторите ввод.

### Как работает валидация под капотом

```
Пользователь вводит сообщение
        ↓
LLM генерирует черновик (draft)
        ↓
draft пустой? → FAIL: "пустой ответ от LLM" → retry
        ↓
validate_draft_against_invariants(client, model, draft, invariants)
  → отдельный LLM-вызов: ожидаем "PASS" или "FAIL: <причина>"
  → неожиданный ответ (не PASS и не FAIL) → FAIL: "validation inconclusive: ..."
  → исключение (сеть, rate limit)          → FAIL: "validation error: ..."
        ↓
  PASS → parse_agent_output(draft) → Response + State Update → пользователю
  FAIL → retry_count += 1
          retry_count < max_retries?
            да → новый LLM-вызов (инварианты уже в системном промпте)
            нет → "[Agent: исчерпаны попытки валидации]" → финальный черновик пользователю
```

В `/plan builder` та же логика, но после `max_retries` провалов вместо молчания builder
генерирует уточняющий вопрос к пользователю, сохраняет **реальный** ответ пользователя
в `plan.clarifications` и продолжает выполнение с расширенным контекстом.

## 9. Исправленные баги (день 15): builder и fail-closed валидация

### Баг 1 — Builder проходил дальше при ошибке валидации

До исправления `validate_draft_against_invariants` возвращала `True` в двух случаях:
- модель вернула что-то кроме PASS/FAIL
- при любом исключении (сеть, rate limit)

Результат: шаг помечался DONE даже если инвариант не прошёл проверку.

```
# БЫЛО (fail-open)
"MAYBE: probably ok"  → passed = True   ← шаг DONE, builder идёт дальше
Exception("timeout")  → passed = True   ← шаг DONE, builder идёт дальше

# СТАЛО (fail-closed)
"MAYBE: probably ok"  → passed = False, reason = "validation inconclusive: 'MAYBE: probably ok'"
Exception("timeout")  → passed = False, reason = "validation error: timeout"
```

---

### Баг 2 — Пустой черновик засчитывался как выполненный шаг

При ошибке API основного LLM `_call_llm_for_builder_step` возвращала `""`.
Пустая строка шла в валидацию, которая из-за бага 1 возвращала `True` → шаг DONE с пустым результатом.

```
# БЫЛО
draft = ""   →  validate(...)  →  True  →  step.status = DONE, step.result = ""

# СТАЛО: проверка ДО валидации
draft = ""   →  passed = False, violation = "пустой ответ от LLM"  →  retry
```

---

### Баг 3 — Фейковые Q&A засоряли `plan.clarifications`

При каждом авто-ретрае (до вопроса пользователю) в `plan.clarifications` добавлялись записи вида:
```json
{"question": "Fix invariant violation: ...", "answer": "Please correct the response..."}
```

Эти записи накапливались и попадали в контекст LLM для следующих шагов, мешая нормальной работе.
Реальные ответы пользователя при этом тонули в шуме.

```
# БЫЛО: после каждого retry
plan.clarifications.append({
    "question": f"Fix invariant violation: {violation}",
    "answer": "Please correct the response to comply with all invariants.",
})

# СТАЛО: просто continue — инварианты уже есть в системном промпте
continue
```

`plan.clarifications` теперь содержит только реальные Q&A от пользователя (после 3 провалов).
