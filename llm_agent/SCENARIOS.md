# Invariant System — Scenarios

**Инвариант** — это ограничение, которое LLM-агент обязан соблюдать в каждом ответе, независимо от запроса пользователя. Список инвариантов задаётся заранее и хранится отдельно от истории диалога. Перед принятием черновика агент проверяет его через отдельный вызов LLM-валидатора: если хотя бы один инвариант нарушен — ответ отклоняется и запрашивается повтор.

---

## Категории инвариантов

### 1. Стек-ограничение

Запрет на использование определённых технологий или библиотек.

**Пример инварианта:** `"/invariant add Хранить данные только в файлах. Не использовать базы данных."`

| Черновик агента | Результат |
|-----------------|-----------|
| "Предлагаю использовать Redis для кэширования." | FAIL |
| "Можно подключить SQLite для хранения истории." | FAIL |
| "Буду сохранять данные в локальный JSON-файл." | PASS |

---

### 2. Техническое решение

Фиксация уже принятого технического решения, которое нельзя пересматривать.

**Пример инварианта:** `"Использовать только стандартную библиотеку для HTTP-запросов. Сторонние клиенты запрещены."`

| Черновик агента | Результат |
|-----------------|-----------|
| "Перепишем код на httpx — он удобнее для async." | FAIL |
| "Предлагаю заменить urllib на requests." | FAIL |
| "Оставляем urllib.request, он уже используется." | PASS |

---

### 3. Бизнес-правило

Требования, продиктованные бизнес-логикой или политиками проекта.

**Пример инварианта:** `"Все ответы агента должны быть на русском языке."`

| Черновик агента | Результат |
|-----------------|-----------|
| "Let me explain the plan step by step." | FAIL |
| "I suggest starting with the data layer." | FAIL |
| "Начнём с уровня данных — это логично." | PASS |

---

### 4. Архитектурное ограничение

Запрет на определённые архитектурные паттерны.

**Пример инварианта:** `"Не использовать глобальное состояние. Все данные передавать явно через параметры."`

| Черновик агента | Результат |
|-----------------|-----------|
| "Добавлю глобальный счётчик вызовов в начало файла." | FAIL |
| "Объявлю модульную переменную `_cache = {}` для переиспользования." | FAIL |
| "Передам счётчик как поле конфигурационного объекта." | PASS |

---

### 5. Противоречивые инварианты

Если заданы взаимоисключающие инварианты, любой черновик нарушит хотя бы один из них.

**Пример:**
1. `"Не добавлять новые классы."`
2. `"Каждый новый функционал оформлять как отдельный класс."`

| Черновик агента | Результат |
|-----------------|-----------|
| "Добавлю класс `Logger` для логирования." | FAIL (нарушает #1) |
| "Добавлю функцию `log_event()` для логирования." | FAIL (нарушает #2) |

Любой черновик будет заблокирован. Это сигнал пользователю пересмотреть список инвариантов.

---

## Поведение при нарушении

### Retry-цикл

При каждом нарушении агент уведомляет пользователя о повторной попытке:

```
[Агент: инвариант нарушен (<причина>). Повтор 1/3...]
[Агент: инвариант нарушен (<причина>). Повтор 2/3...]
```

Инварианты присутствуют в системном промпте, поэтому retry происходит без инъекции фейковых сообщений в историю диалога.

### Исчерпание попыток

После исчерпания лимита повторов запрос отклоняется, черновик не включается в историю:

```
[Агент: исчерпаны попытки валидации. Последнее нарушение: <причина>]
```

### Уточняющий вопрос

Если автоматические повторы не помогают, агент запрашивает уточнение у пользователя:

```
[Агент: уточняющий вопрос]
Как выполнить шаг без нарушения инварианта "<инвариант>"?
```

Ответ пользователя добавляется в контекст шага. Если после уточнения нарушение повторяется снова — шаг помечается как невыполнимый.

---

## Fail-closed принцип

При любой неопределённости — считать, что инвариант нарушен:

| Ответ LLM-валидатора | Решение |
|----------------------|---------|
| `"PASS"` | Принять черновик |
| `"FAIL: <причина>"` | Отклонить, показать причину |
| Любой другой текст | Отклонить (неопределённость = нарушение) |
| Ошибка API / таймаут | Отклонить (сбой = нарушение) |
| Пустой черновик от агента | Отклонить без вызова валидатора |

Правило: сомнение разрешается в пользу инварианта, а не в пользу принятия черновика.

---

## Персистентность инвариантов

Инварианты хранятся **отдельно от истории сообщений** — в специальном разделе состояния сессии. Это принципиально:

- история диалога очищается / обрезается при смене контекста
- инварианты сохраняются на протяжении всей сессии и восстанавливаются при её возобновлении

При сохранении сессии инварианты должны сериализоваться как самостоятельный список, не перемешиваясь с сообщениями пользователя и агента.

# Сценарии ручной проверки FSM задачи

Каждый сценарий запускается с чистого старта:

```bash
source .venv/bin/activate
python script.py
```

## Граф допустимых переходов

```
PLANNING ──► EXECUTION ──► VALIDATION ──► DONE
                │  ▲            │
                │  └────────────┘  (возврат на доработку)
                ▼
             PAUSED ──► EXECUTION
                │
                ▼
             FAILED  (терминальное)
DONE             (терминальное)
```

---

## Сценарий 1. Основной цикл: PLANNING → EXECUTION → VALIDATION → DONE

```
/task new Написать отчёт за март по биткоину
```
Ожидаем: план создан, фаза `planning`, список шагов.

```
/task start
```
Ожидаем: фаза → `execution`.

```
/task show
```
Ожидаем: `Phase: execution`.

```
/task step done Черновик написан
```
Ожидаем: `[Шаг 1 завершён]`. Повторить для каждого шага. После последнего шага фаза автоматически → `validation`.

```
/task show
```
Ожидаем: `Phase: validation`.

```
/task done Отчёт сдан в архив
```
Ожидаем: фаза → `done`, `active_task_id` сбрасывается.

---

## Сценарий 2. Пауза и возобновление

**Что проверяем:** пауза разрешена только из EXECUTION; из PAUSED нельзя выполнять шаги.

```
/task new Настроить CI-пайплайн
/task start
/task pause
```
Ожидаем: фаза → `paused`.

```
/task step done что-то
```
Ожидаем: **ошибка** `[Команда step доступна только в фазе execution. Текущая: paused]`.

```
/task resume
```
Ожидаем: фаза → `execution`.

```
/task step done готово
```
Ожидаем: шаг завершён, работа продолжается.

---

## Сценарий 3. Пауза → выход → возобновление сессии

**Что проверяем:** `active_task_id` и фаза `paused` переживают перезапуск.

```
/task new Мигрировать базу данных
/task start
/task pause
```

Нажмите **Ctrl+C**.

```bash
python script.py --resume
```

```
/task show
```
Ожидаем: та же задача, фаза `paused`.

```
/task step done попытка
```
Ожидаем: **ошибка** — step доступна только в фазе execution.

```
/task resume
/task step done миграция применена
/task done база перенесена
```
Ожидаем: фаза → `done`.

---

## Сценарий 4. Недопустимые переходы

### 4a. `done` из EXECUTION (минуя validation)

```
/task new Тест прыжка
/task start
/task done пробуем
```
Ожидаем: **ошибка** `[Переход execution → done недопустим]`. Фаза остаётся `execution`.

### 4b. `pause` из PLANNING

```
/task new Тест паузы из планирования
/task pause
```
Ожидаем: **ошибка** о недопустимом переходе.

### 4c. `resume` без паузы

```
/task new Тест resume
/task start
/task resume
```
Ожидаем: **ошибка** — resume разрешён только из `paused`.

### 4d. Любая команда после DONE

```
/task new Финал
/task start
/task step done шаг выполнен
/task done задача закрыта
/task pause
/task resume
/task fail
```
Ожидаем: все три команды выдают ошибку — `done` терминальное.

---

## Сценарий 5. Провал задачи (FAILED)

```
/task new Задача которая провалится
/task start
/task fail что-то пошло не так
```
Ожидаем: фаза → `failed`, `active_task_id` сбрасывается.

```
/task resume
```
Ожидаем: **ошибка** — из `failed` нельзя возобновить.
